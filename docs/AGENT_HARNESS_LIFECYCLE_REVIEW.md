# Agent Harness & Lifecycle Review

**Date:** 2026-07-11 · **Scope:** the LLM agent harness, the appointment lifecycle FSM + transition side effects, and the background runners that drive the lifecycle — reviewed as one end-to-end system.

## How this review was run

A multi-agent review harness produced this document: one mapping agent traced the runtime interactions; five parallel reviewers examined the system through distinct lenses (concurrency/idempotency, error handling & crash recovery, duplication/consolidation, agent-loop correctness, architecture/boundaries); findings were deduplicated across lenses; then **every finding was adversarially verified** — an independent agent attempted to refute each one against the actual code, and high-severity findings additionally received a severity audit. Finally three agents produced competing refactor proposals from different angles (incremental evolution of `docs/REFACTOR_PLAN.md`, kernel-first restructure, lifecycle-authority rearchitecture), which section 3 synthesizes.

**Result: 33 findings confirmed (5 high, 18 medium, 10 low), 1 refuted.** No finding in this document survived without an independent verification pass. Severity is calibrated to the real deployment: single instance, ~100 appointments/day, with genuine in-process concurrency (concurrent HTTP requests, Gmail Pub/Sub push + 3-minute polling fallback, BullMQ workers, 10+ periodic timers).

## 1. Executive summary

The system is in better shape than a 14,000-line review scope usually yields: the lifecycle FSM migration (`domain/scheduling/lifecycle/`) is real and nearly watertight — atomic status preconditions, generation counters, and audit trails are consistently applied, and the one suspected raw-status-write bypass was **refuted** on verification (§6). The side-effect tracker/harness/retry stack shows careful thinking about idempotency, leases, and replay.

The defects cluster in three places, and they are structural, not scattered:

1. **The retry path is a second, divergent implementation of every side effect.** Each sentinel-gated effect exists twice: an inline closure at the call site (send → confirm sentinel → checkpoint → lifecycle transition) and a hand-mirrored per-effect-type branch in `side-effect-retry.service.ts` that **replays only the email half**. On any retried effect, the sentinel confirm and the `feedback_requested` transition are silently dropped (HIGH). The register-after-commit crash window, the stale-generation double-send, and the chase generation-stranding are all symptoms of the same split-brain design.
2. **The agent turn has last-writer-wins state loss and several dishonest success paths.** A `ConcurrentModificationError` on checkpoint save is "resolved" by adopting the other writer's version and continuing — silently discarding one turn's conversation state (HIGH). The email-loss compensation net that was supposed to catch exactly this filters on tool names the agent never produces, so it is dead code (HIGH). `startScheduling` and the send tool both swallow terminal failures and report success, wedging appointments in states no runner recovers.
3. **The kernel boundary was populated by god-module lineage, not by the kernel rules.** `core/agent/tools/` and `core/email/inbound/` are the scheduling product's brain sitting inside the "generic kernel" (both HIGH): handlers import `domain/scheduling/lifecycle`, the inbound router imports appointment matching and closure logic. The ESLint boundary rule the plan promised was never added — no ESLint config exists at all — so ~40 violating import edges accumulated silently.

Alongside these, a consistent duplication tax: two agent loops with six hand-mirrored escalation blocks, two Redis tool-idempotency layers with **opposite** failure semantics (fail-closed vs fail-open), five background services re-implementing `LockedPeriodicService`, chase-email hand-rolling `sentinel-batch-runner`, and stuck-sentinel recovery implemented twice with inline `new Date(0)` literals.

## 2. Verdict on `docs/REFACTOR_PLAN.md`

The review was asked to challenge the in-flight plan, and the challenge lands in three specific places. The plan's *goals* (kernel boundary, god-module splits, manageable PRs, ATS readiness) survive intact; its *sequencing and two placement decisions* do not:

1. **The plan has no hardening track, and defects don't follow module geography.** The plan sequences by file layout (availability next, then schema surgery), but the highest-severity live bugs — half-replayed retries, last-writer-wins saves, always-throwing recovery SQL — cut across modules and outrank every remaining structural phase. A hardening track must come first.
2. **Phases 2b/2c put policy in the kernel; the placement should be reversed.** `core/agent/tools/` and `core/email/inbound/` fail all three of `core/README.md`'s own rules. The *mechanism* (loop, dispatch pipeline, idempotency, transport) belongs in `core/`; the *policy* (tool handlers, inbound routing, prompts) belongs in `domain/scheduling/`. This is a correction of placement, not of intent — the plan's ATS-lift goal is exactly why the correction matters.
3. **Reliability work was mis-filed under deferred "multi-instance readiness."** Registering side effects inside the transition transaction is not scaling work — it is single-instance crash/deploy safety (a SIGKILL between commit and registration currently drops confirmation emails with no trace). The plan's own expand/dual-write/contract discipline (Phase 3) is the right tool; it just needs to be applied to `side_effect_logs`.

Two smaller corrections: the plan document itself is stale (Phase 1c is described as "next PR" but has already landed; `lifecycle/README.md` points at a deleted file), and the "enforced by code review, not yet by lint rules" stance has demonstrably failed — the lint rule needs to land now, allowlist-first, not after the tree is clean.

## 3. Recommended refactor plan

Synthesized from the three competing proposals (appendices in `docs/agent-harness-review/`). All three independently converged on: hardening before any moves, rebuilding the effects layer as a register-in-transaction outbox with a single handler path, consolidating runner scaffolding, and reversing the Phase 2b/2c kernel placement — that convergence is strong evidence for the backbone. Where they diverge, this plan takes the lifecycle-authority proposal's effects design (evolve `side_effect_logs` in place; no new table), the incremental proposal's fix granularity, and the kernel-first proposal's boundary endgame — while **deferring** its riskiest leg (unifying the two agent loops) and **rejecting** the declarative-rules-engine leg (stale-check's 982 lines are essential business policy, not scaffolding; a DSL rich enough to express it would be a worse programming language than TypeScript).

Each stage is one or more single-PR units. Existing plan Phases 3b/3c (AppointmentConversation cutover) continue in parallel, sequenced after Stage A2 because they flip the same state-save writers.

### Stage A — Hardening (3 PRs, zero file moves)

- **A1 — Dead and broken recovery paths** *(very low risk, ship immediately).* Fix the three raw-SQL sites that query the Prisma model name `"AppointmentRequest"` instead of the mapped table `appointment_requests` — all three currently throw on every invocation (`email-queue.service.ts:402`, `core/email/outbound/queue.ts:241`, `core/email/inbound/divergence-handling.ts:102`). Fix the email-loss compensation filter from `send_user_email|send_therapist_email` to `send_email` (`ai-conversation.service.ts:471`, `justin-time.service.ts:162`). Make `startScheduling` propagate final state-save failure instead of returning success, so the `justintime_start` outbox row is re-driven; consolidate its inline retry loop onto `storeConversationStateWithRetry` (make `expectedUpdatedAt` optional). Verification: real-Postgres `$executeRaw` test (mock-DB tests are why the table-name bugs survived); compensation-path unit test; existing `justin-time-outbox.test.ts`.
- **A2 — Mid-turn races** *(medium risk; the hot path).* (a) Per-appointment Redis turn mutex (TTL + renewal, same pattern as the existing message lock) around `processEmailReply`/`startScheduling` — at this volume contention is rare and the second turn *should* see the first's state. (b) With turns serialized, `ConcurrentModificationError` becomes a genuine anomaly: stop adopting the fresh version at `justin-time.service.ts:623`; abort the turn and leave the Gmail message unmarked so the scanner replays it. (c) `processEmailReply` returns failure when the final save fails. (d) Add `status: { notIn: TERMINAL_STATUSES }` to the pre-send gates (`core/agent/tools/send.ts:117`, `dispatch.ts:104`) — closes the mid-turn-cancel send window. (e) Break the tool loop when a tool skips with `human_control`, and make `flagForHumanReview` preserve an existing takeover record. Roll the mutex out behind a setting. Verification: two-concurrent-turns test asserting the loser aborts unmarked; mid-turn-cancel test asserting send refusal; `agent-tool-loop-safety-circuits.test.ts`, `paused-message-not-marked.test.ts`.
- **A3 — Behavioral traps** *(low risk).* Chase passes its claim epoch as `scopeGeneration` (re-armed chases currently dedupe against dead rows and never fire). Stage-gated tools stop stripping `mark_scheduling_complete` in the stages direct-link bookings confirm from (`tools-for-stage.ts:25`). The send fallback enqueues the tracking-coded subject and known threadId, and the queue drain stamps new Gmail thread IDs back (lift the stamping block from `send.ts:183-244` into a shared helper). When direct send *and* enqueue both fail, the tool reports failure and is not idempotency-marked; flag for human review. Decide the prompt-injection signal's fate: gate something (forced review / tightened budget) or delete the dead branch — don't leave it looking handled. Add a per-turn nonce to the idempotency hash for repeatable canned sends.

### Stage B — Effects layer rebuilt as an outbox (2 PRs; the defect-dense seam)

- **B1 — Register-in-tx + handler registry for transition effects** *(highest-risk PR of the plan).* Add a `params` JSONB column to `side_effect_logs` (expand-only). Transition transactions (`terminal-tx.ts`, `confirmed.ts`; light transitions wrap `updateMany` + insert in an interactive tx) insert effect-intent rows **before commit**, closing the crash window at `confirmed.ts:330`. Build `lifecycle/effects/`: one handler per effect type (bodies lifted verbatim from `transition-side-effects.service.ts`, `appointment-notifications.service.ts`, and the retry branches), a drain runner (renamed `side-effect-retry` on `LockedPeriodicService`, 30s cadence + post-commit in-process poke so user-visible latency stays ~0). Payload rendered at first drain and persisted — the render-once/replay-verbatim protection is kept. Keep legacy-format rows drainable during rollout; retention already ages the table at 24h/terminal. This PR folds in the plan's deferred move of `transition-side-effects.service.ts` + `appointment-notifications.service.ts` into `domain/scheduling/lifecycle/` — the registration ordering is being rewired anyway.
- **B2 — Sentinel-gated periodic effects onto the outbox** *(fixes the HIGH half-replay class).* Chase, meeting-link check, session reminders, feedback dispatch, feedback reminder, therapist nudge: sentinel claim + intent-row insert become one transaction; the handler owns the FULL sequence (state re-check → send → `confirmSentinelClaim` → checkpoint / `transitionToFeedbackRequested`) and runs **identically on first drain and retry** — the per-effect-type retry branches in `side-effect-retry.service.ts` are deleted, so retries can no longer drop the finalization half. `registerInTx` supersedes older-generation pending/failed rows for the same (scope, effectType), fixing the stale-generation double-send. Rewrite `side-effect-retry-executor.test.ts` against the registry *in the same PR* — it currently asserts the buggy split design — and add the previously-missing assertions (sentinel confirmed + transition fired on the retry path), plus a kill-between-commit-and-drain test.

### Stage C — Scaffolding consolidation (1–2 PRs, mostly mechanical deletion)

Migrate the five hand-rolled interval+lock services (therapist-nudge, missed-message-scanner, pending-email, slack-weekly-summary, work-report) and the bare Slack `setInterval` onto `LockedPeriodicService`; move `postBookingFollowupService` onto it too (it is the only appointment-mutating runner with no lock). Fix `LockedTaskRunner`'s timeout semantics: on timeout, alert but do not release the lock or clear the overlap guard until the underlying promise settles (today's release lets a zombie task overlap its successor). Chase-email migrates onto `sentinel-batch-runner`. `atomic-sentinel-claim.ts` gains `EPOCH_SENTINEL` and `cleanupStuckSentinels(fields, olderThan)`; every inline `new Date(0)` literal dies. Shared constants: `TERMINAL_STATUSES` at the inline-literal sites (`stale-check.service.ts:326,478`, `admin-force.ts:187,201`) and a named `ACTIVELY_NEGOTIATING_WHERE` fragment. Agent-side dedup in the same stage: extract the six-times-mirrored escalation block into `tool-loop-helpers.ts`; delete the availability agent's fail-open idempotency copy and parameterize the canonical fail-closed module by key prefix; share `normalizeAgentOutboundEmail()` (subject prefix + body normalization) between both agents; make `startReschedulingState` the single enter-rescheduling writer.

### Stage D — Boundary re-draw + enforcement (3 PRs)

- **D1 — Lint rule now, allowlist-first** *(near-zero risk).* Create the ESLint config (none exists despite the `lint` script) with `import/no-restricted-paths` banning `core/** → domain/**` and `core/** → scheduling services`, with an explicit exceptions file naming today's ~40 violating edges. New violations stop immediately; the allowlist shrinks with each move.
- **D2 — `core/agent/tools/` → `domain/scheduling/agent/`.** Handlers, dispatch, send, and the `schedulingTools` schema block (moved out of `agent-tool-loop.ts` beside its dispatch) are scheduling policy; a slimmed `core/agent/` keeps the genuinely generic mechanism (idempotency, email normalization, escalation, the loop itself). Pure `git mv` + import rewrite, Phase-1c pattern; `tsc` is the real gate.
- **D3 — `core/email/inbound/` routing → `domain/scheduling/inbound/` + shim dissolution.** Appointment matching, weekly-mailing, nudge-reply, closure-auto-dismiss, unmatched-attempts move to domain; transport (outbound send, queue drain, dedup facade, lock renewal, MIME) stays in `core/email/`. The `agent-processor.ts` DI registry dissolves — the domain can import its own agent directly. Dissolve the `email-processing.service.ts` backward-compat shim (the migration policy forbids kept shims). Then tighten the lint rule to strict, refresh `core/README.md` + `lifecycle/README.md`, and update `REFACTOR_PLAN.md` to record this findings-driven replan.

### Explicitly deferred / rejected

- **Two-loops-into-one (kernel-first Phase 4) — deferred.** After Stage C the loops share escalation, idempotency, and normalization; what remains distinct is checkpoint hooks, terminal tools, and guard chains. Unification's payoff is real but its risk (subtle LLM-path behavior drift that tests can't fully capture) is the highest in any proposal, and the kernel-first author flagged it as the phase to abandon first. Revisit when the ATS agent's real requirements are known — that's the moment a `ToolTable` registry design earns its cost.
- **Declarative rules engine for background services — rejected** (see above).
- **Multi-instance readiness — still out of scope**, unchanged from the plan.

### Sequencing rationale

Stage A fixes live bugs with no structural risk. Stage B is the surgery, sequenced early because four confirmed findings share its root cause; B1 before B2 so the compatibility window exists. Stage C is safe mechanical deletion that shrinks the surface Stage D has to move. Stage D last because moves are cheapest when the code being moved is already consolidated — and D1 (the lint rule) lands first within the stage so the boundary stops eroding while the moves proceed. Total: ~10 PRs. If capacity forces triage: A1, A2, and B2 carry the highest defect density; D can wait indefinitely without correctness cost (but blocks the ATS lift).

### 3. Side-effect flow and guarantees

Transition commits, then side effects are **fire-and-forget** (`lifecycle/dispatch-helpers.ts:33 fireAndForget`) — never awaited, never roll back the transition:
- transition → `transitionSideEffectsService.onConfirmed/onCancelled/onCompleted` (transition-side-effects.service.ts) + `appointmentNotificationsService.notify*` → both wrap work in `side-effect-harness.ts` wrappers → `side_effect_logs` table via `side-effect-tracker.service.ts` → failures re-driven by `side-effect-retry.service.ts` (LockedPeriodicService, 5 min).
- Harness sequence (side-effect-harness.ts:62): register (idempotency key = hash(scopeId, transition, effectType[, transitionGeneration])) → skip if completed/abandoned → **CAS execute-lease claim** `tryClaimEffect` (:107, prevents retry-runner double-fire) → execute → markCompleted / markFailed.
- Guarantees: **at-least-once with idempotency-key dedup** for tracked effects; replay-sensitive emails persist a rendered payload at registration and replay verbatim (`runReplayableTrackedSideEffect` :196; retry never re-renders — side-effect-retry.service.ts:335-364). Registration failure degrades to untracked fire-once (:70-79). Retry has attempts budget → `abandoned` + Slack alert (side-effect-retry.service.ts:128-152). SSE (`notifyTransition`) and some Slack notifications are pure fire-and-forget with no durability. The generation scoping means cancel→re-confirm doesn't dedupe against the previous generation's rows.
- **Outbox proper** exists in exactly one place: `justintime_start` (appointments.routes.ts:467, registered pre-commit).
- Harness explicitly does **not** provide mutual exclusion for periodic effects (side-effect-harness.ts:250-255) — callers must claim a sentinel first (see §5).

#### 2. Checkpoint save on ConcurrentModificationError refreshes the version and continues, converting the conversation-state optimistic lock into last-writer-wins across concurrent agent turns

Nothing serializes agent turns per appointment: the email pipeline's Redis lock is per Gmail message (core/email/inbound/process.ts:115-116), and a user reply and a therapist reply (or push vs poll delivering two different messages) can run two concurrent processEmailReply turns against the same appointment. **Evidence:** justin-time.service.ts:619-628 'Checkpoint COMod — skipping intermediate save, refreshing version for the final save' followed by `currentStateVersion = updated?.updatedAt ?? new Date()`; :654-659 final save uses that refreshed version; :660-684 returns `{ success: true, message: 'Email processed and response sent' }` even when `saveResult.success === false` ('Final state save failed after all retries - compensation recorded'). ai-conversation.service.ts:450-455 shows the final save intentionally does NOT retry on COMod ('they indicate a real conflict') — a protection the refreshed version sidesteps.
