# Incremental Evolution Proposal — therapist-scheduler backend

Backbone: `docs/REFACTOR_PLAN.md` stays authoritative. Findings slot in as a new **Hardening track (H)** that lands *before* the plan's Phase 3b cutover, a **Consolidation track (C)** that absorbs the duplication findings, and a **Boundary track (B)** that corrects where Phases 2b/2c placed scheduling code in `core/`. Every phase is one reviewable PR, behavior-preserving unless the finding says the behavior is a bug.

## 3. Divergences from REFACTOR_PLAN.md, and why

1. **New H-track precedes Phase 3b.** The plan has no hardening track; the two HIGH runtime findings (retry-half-replay, last-writer-wins) are live data-loss bugs and outrank schema cutover.
2. **Phase 2c/2b placement partially reversed.** The plan put the tool executor and inbound pipeline in `core/`; the boundary findings show they fail all three kernel rules. B2/B3 move them to `domain/scheduling/`. `core/` keeps only what genuinely survives an ATS lift — the plan's *goal* is preserved, its Phase-2 *placement* is corrected.
3. **Lint rule timing.** Plan said "add in the Phase 1c PR"; that fired without the rule and ~40 violating edges accumulated. Diverge to a two-step: allowlisted rule immediately (B1), strict rule after the moves (B4). Waiting for a clean tree first (the plan's implicit stance) would let violations keep growing.
4. **`transition-side-effects` / `appointment-notifications` move promoted** from "low-priority future small PR" into H7, because H7 rewires their registration ordering anyway — doing move and rewire separately doubles review of the same lines.
5. **`email-processing.service.ts` facade gets a scheduled dissolution (B3)** instead of the plan's indefinite "aligning is a future PR" — it is the remaining two-name send path and a core→services→core knot.
6. **`postBookingFollowupService` locking** was not in the plan at all; C1 adds it because it is the only appointment-mutating runner relying solely on sentinels.

## 4. Honest costs / risks of this angle

- **~15 PRs, long tail.** Incrementalism defers the boundary payoff: `core/agent` and `core/email/inbound` files move *twice* in git history (into core in Phase 2b/2c, out again in B2/B3), churning blame and review attention. A design-first rewrite would move them once.
- **Behavior-changing hardening under live traffic.** H3 (returning `success:false`, failing the losing turn) and H5 (tool reporting failure instead of silent success) change caller-visible semantics; a missed caller assumption can turn a silent bug into a visible outage. Mitigation is the setting-gated rollout in H3 and the per-sub-fix tests in H5, but the risk is real and is the price of fixing bugs in place.
- **H2's dual payload format** (old send-only vs new finalize-carrying rows) is transitional complexity that lives in production for weeks and must itself be tested.
- **The mutex in H3 adds serialization** on the agent hot path; a stuck turn (Claude latency) delays the next inbound reply for the same appointment until TTL expiry. Acceptable at this volume, but it is new queueing behavior.
- **The plan-as-backbone keeps some sunk decisions**: e.g. `SchedulingContext`-typed dispatch stays scheduling-typed after B2 (legal in domain/) rather than being redesigned generic — the ATS agent will need its own dispatch or a later generalization PR, which this proposal defers rather than solves.
- **Ordering couples tracks**: H7 depends on H2, 3b depends on H3/H4, B-track depends on C3 having moved `schedulingTools`. A stall mid-track (staffing, incident) leaves the tree in a mixed state — coherent at every commit, but with `REFACTOR_PLAN.md` needing continuous upkeep to stay truthful (the stale-plan finding shows this team has already dropped that ball once).
